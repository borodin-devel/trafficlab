# Проект дерева архитектурной документации

## Цель

Разделить содержимое корневых файлов `ARCHITECTURE.md` и `CAPTURE.md` на связанное дерево документов в каталоге `architecture/`.

Основной путь чтения должен повторять стадии рабочего конвейера. Общие правила, контракты и взаимозаменяемые реализации должны храниться отдельно от стадий. Один факт должен иметь один основной документ.

После полного переноса корневые файлы `ARCHITECTURE.md` и `CAPTURE.md` удаляются. Файлы-указатели на их месте не создаются.

## Основные принципы

1. Стадии конвейера являются главным способом навигации.
2. Общие правила не копируются в документы стадий.
3. Форматы обмена описываются как отдельные контракты.
4. Взаимозаменяемые реализации одного типа располагаются на одном уровне дерева.
5. Нумерация стадий использует шаг 10. Это позволяет вставлять новые стадии без переименования существующих.
6. Имена остальных каталогов и файлов не содержат порядковых номеров.
7. Каждый самостоятельный документ раскрывает термин или сокращение при первом употреблении.
8. Текст использует простой, сухой и понятный язык.

## Назначение верхних разделов

### `README.md`

Содержит карту дерева, рекомендуемый порядок чтения и правила размещения новых документов. Не пересказывает содержание дочерних файлов.

### `system/`

Описывает систему целиком: границы, общий поток данных, программы, внутренние модули и поддерживаемые сценарии запуска.

### `stages/`

Описывает конвейер по порядку выполнения. Каждая стадия определяет своё назначение, входы, выходы, процесс и инварианты. Стадия ссылается на общие контракты и доступные реализации.

### `contracts/`

Описывает стабильные файловые форматы и структуры данных между программами и стадиями. Контракты не зависят от конкретных инструментов или алгоритмов.

### `implementations/`

Описывает конкретные взаимозаменяемые реализации. Все реализации одного контракта находятся в одном каталоге и на одном уровне.

### `common/`

Описывает правила, действующие для нескольких стадий: жизненный цикл артефакта, воспроизводимость, оркестрацию, ресурсы, наблюдаемость, безопасность и платформенные ограничения.

### `project/`

Описывает структуру репозитория, зависимости, стиль программирования, тестирование, хранение крупных файлов, порядок разработки, архитектурные решения и риски.

## Стандарт документа стадии

Каждый файл `stages/*/README.md` использует одинаковые разделы:

1. Назначение.
2. Место в конвейере.
3. Входы.
4. Выходы.
5. Основной процесс.
6. Инварианты.
7. Ошибки и восстановление.
8. Используемые контракты.
9. Доступные реализации.
10. Предыдущая и следующая стадии.

Подробный файл внутри каталога стадии используется только тогда, когда тема слишком велика для `README.md` или имеет самостоятельную границу ответственности.

## Границы документов захвата

- `stages/10-capture/README.md` хранит назначение, входы, выходы и требования `CAP-*`.
- `lifecycle.md` хранит порядок подготовки, запуска, остановки и очистки.
- `network-environment.md` хранит общие требования к сетевой среде стадии: адресацию, маршрутизацию, Domain Name System и ограничения Windows Subsystem for Linux.
- `process-control.md` хранит правила control group version 2, запуска процесса, сигналов и терминального интерфейса.
- `packet-capture.md` хранит общие требования к готовности захвата, интерфейсам, направлению пакетов, длине захватываемой части пакета и завершению записи.
- `publication.md` хранит состав результата, первичную проверку и условия публикации.
- `implementations/capture-backends/netns-dumpcap.md` хранит конкретную реализацию через сетевое пространство имён Linux — Linux network namespace, пару виртуальных Ethernet-интерфейсов и `dumpcap`.
- `ebpf-cgroup.md` и `container.md` хранят рассмотренные альтернативы на том же уровне.

Общие требования стадии не повторяются в документах реализаций. Документ реализации содержит только её устройство, ограничения и отличия.

## Границы документов моделей

- `stages/40-train/README.md` хранит общий процесс обучения и правила выбора модели.
- `contracts/model-artifact.md` хранит единый формат результата обучения.
- `implementations/models/` содержит отдельный равноправный файл для каждого типа модели.
- Общие требования `MOD-*` не копируются в документы отдельных моделей.

Такое же разделение применяется к реализациям захвата, декодерам пакетов, средствам построения пакетов, извлекателям признаков, генераторам, метрикам, валидаторам, оптимизаторам и сериализаторам артефактов.

## Правила ссылок

Связи между разделами имеют следующее направление:

```text
system/pipeline.md
    -> stages/*

stages/*
    -> contracts/*
    -> implementations/*
    -> common/*

implementations/*
    -> contracts/*
    -> использующие реализацию stages/*
```

Файлы `contracts/*` не ссылаются на конкретные реализации.

Все внутренние ссылки являются относительными. Ссылка ведёт на основной документ темы, а не на копию текста.

## Владение информацией

- Порядком конвейера владеет `system/pipeline.md`.
- Форматом данных владеет соответствующий файл в `contracts/`.
- Процессом стадии владеет соответствующий каталог в `stages/`.
- Поведением конкретного инструмента или алгоритма владеет файл в `implementations/`.
- Общесистемным правилом владеет файл в `common/`.
- Устройством исходного кода и разработки владеет файл в `project/`.

Другой документ может дать короткий контекст и ссылку. Полный повтор определения, перечня полей, алгоритма или требования не допускается.

## Требования и идентификаторы

Идентификаторы `CAP-*`, `DAT-*`, `MOD-*`, `GEN-*`, `CMP-*` и `EVO-*` сохраняются.

Требование хранится в документе владеющей стадии. Другие документы ссылаются на его идентификатор. Нефункциональное требование, действующее для нескольких стадий, хранится в соответствующем документе `common/`.

## Статус реализаций

Каждый файл в `implementations/` начинается с метаданных:

```text
Статус: основная | дополнительная | будущая
Контракт: относительная ссылка
Используется стадиями: относительные ссылки
```

Это отделяет выбранную реализацию от альтернатив и будущих вариантов.

## Язык документов

Документы пишутся по-русски. Предложения должны быть короткими. Один абзац должен выражать одну мысль.

Термин раскрывается при первом употреблении в каждом самостоятельном документе. Английское название сохраняется, если оно нужно для поиска документации, настройки инструмента или сопоставления с кодом.

Пример:

```text
Ориентированный ациклический граф — Directed Acyclic Graph (DAG).
```

После раскрытия используется сокращение. Нераскрытый профессиональный жаргон не используется.

## Перенос существующего содержания

Содержимое `ARCHITECTURE.md` распределяется между всеми верхними разделами дерева. Содержимое `CAPTURE.md` распределяется главным образом между `stages/10-capture/`, `implementations/capture-backends/netns-dumpcap.md`, документами проверки и платформенными правилами.

Повторяющиеся фрагменты двух исходных файлов объединяются. Более подробная и непротиворечивая формулировка становится основной. Остальные места получают ссылку.

Внешние источники из `CAPTURE.md` переносятся в раздел ссылок того документа, где используется подтверждаемое ими утверждение.

После переноса выполняется проверка, что содержательные разделы исходных файлов имеют нового владельца. Затем исходные файлы удаляются.

## Критерии готовности

Работа считается завершённой, когда выполнены все условия:

1. Создано согласованное дерево `architecture/`.
2. Все содержательные разделы `ARCHITECTURE.md` и `CAPTURE.md` перенесены.
3. Повторяющиеся определения и требования объединены.
4. Внутренние относительные ссылки разрешаются.
5. Стадии связаны в порядке конвейера.
6. Взаимозаменяемые реализации одного типа находятся на одном уровне.
7. Термины и сокращения раскрыты при первом употреблении.
8. Корневые `ARCHITECTURE.md` и `CAPTURE.md` удалены.
9. Корневые файлы-указатели не созданы.
10. `architecture/README.md` позволяет найти любой основной раздел.
