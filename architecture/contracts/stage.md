# Контракт состояния стадии

## Назначение

`stage.json` фиксирует состояние одного выполнения стадии. Файл является источником истины о готовности результата, даже если остальные файлы стадии уже существуют.

Версия контракта задаётся полем `schema_version`. Начальная версия — `1.0.0`; она изменяется по правилам семантического версионирования.

## Обязательные поля

```json
{
  "schema_version": "1.0.0",
  "stage_type": "convert",
  "stage_key": "sha256:...",
  "status": "completed",
  "created_at": "2026-07-13T12:34:56.000000Z",
  "started_at": "2026-07-13T12:34:57.000000Z",
  "finished_at": "2026-07-13T12:35:30.000000Z",
  "normalized_config": {},
  "ordered_input_hashes": ["sha256:..."],
  "implementation_version": "...",
  "output_hashes": {"dataset": "sha256:..."},
  "error": null
}
```

`status` принимает значения `created`, `running`, `completed`, `failed`, `cancelled` или `invalid`. `started_at` может быть `null` для `created`. `finished_at` обязано быть `null` для `created` и `running`. `error` может быть объектом с машиночитаемым кодом и сообщением либо `null`.

Ключ вычисляется детерминированно:

```text
stage_key = hash(
    stage_type,
    normalized_config,
    ordered_input_hashes,
    implementation_version
)
```

## Инварианты

- Только `status = completed` разрешает использовать выходы как вход другой стадии.
- `completed` допустим только после проверки всех заявленных выходов.
- Порядок `ordered_input_hashes` участвует в вычислении ключа и не нормализуется сортировкой.
- Одинаковые входы, конфигурация и версия реализации дают одинаковый `stage_key`.
- Повторное выполнение не изменяет опубликованный результат с тем же ключом.
- Временные метки записываются в Coordinated Universal Time (UTC) в формате Request for Comments (RFC) 3339.

## Потребители

Контракт читают оркестратор, механизм кэша, команды возобновления и инспекции, а также каждая следующая стадия. Правила публикации описаны в [жизненном цикле артефакта](../common/artifact-lifecycle.md).
