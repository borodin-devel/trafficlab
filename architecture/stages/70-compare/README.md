# Сравнение наборов данных

## Назначение

Стадия измеряет сходство исходного и синтетических наборов сетевого трафика. Она сохраняет результаты отдельных метрик и формирует многокритериальную оценку, не сводя качество к одному числу.

## Место в конвейере

Стадия выполняется после [проверки синтетического трафика](../60-validate-synthetic/README.md). Её результат можно использовать для отчётности или передать [эволюционному циклу](../80-evolve/README.md).

## Входы

- один исходный dataset package — пакет нормализованного набора данных — по [контракту набора данных](../../contracts/dataset.md);
- один или несколько синтетических пакетов по тому же контракту;
- успешные отчёты проверки всех входных пакетов;
- профиль сравнения с выбранными семействами метрик, параметрами и правилами агрегации.

Стадия не читает Packet Capture Next Generation (PCAPNG) напрямую. Исходный и синтетические пакеты должны иметь совместимые версии схемы, единицы измерения и определения направлений трафика.

## Выходы

Стадия публикует `metrics.parquet`, `comparison.json` и `report.md` по [контракту результата метрики](../../contracts/metric-result.md). Первые два файла предназначены для программной обработки, третий — для чтения человеком.

Результат содержит отдельное сравнение для каждого синтетического пакета. Записи `results[]` сохраняются в каноническом порядке синтетических входов, описанном ниже.

## Основной процесс

Стадия проверяет совместимость пакетов и отчётов валидации. До вычисления `stage_key` она канонически сортирует синтетические входы по `artifact_id`. Исходный пакет и его отчёт занимают фиксированные позиции, а синтетические пакеты и связанные с ними отчёты добавляются в порядке этой сортировки при формировании `ordered_input_hashes` из [контракта состояния стадии](../../contracts/stage.md). Контракт не сортирует `ordered_input_hashes` автоматически: порядок участвует в ключе, поэтому именно это правило стадии делает ключ независимым от порядка перечисления синтетических входов при вызове.

Затем стадия выбирает непересекающиеся по целым сессиям обучающие и тестовые части для метрик, которым требуется обучение. Выбранные реализации вычисляют отдельные результаты для каждого синтетического пакета. Записи `results[]` сохраняются в том же порядке `artifact_id`, в котором синтетические входы вошли в `ordered_input_hashes`.

После вычисления стадия формирует [многокритериальную оценку](evaluation.md), записывает машиночитаемые файлы и строит человекочитаемый отчёт. Ошибка одной метрики сохраняется как диагностируемый результат и не подменяется числом успешной метрики.

## Инварианты

Score — числовая оценка качества. Индивидуальные оценки и диагностика остаются доступны независимо от наличия сводной оценки.

| ID | Требование |
| --- | --- |
| CMP-001 | Сравнение работает с нормализованными dataset package, а не непосредственно с `.pcapng`. |
| CMP-002 | Метрики группируются по семействам. |
| CMP-003 | Каждая метрика возвращает значение, направление оптимизации и диагностику. |
| CMP-004 | Сводный score не заменяет индивидуальные метрики. |
| CMP-005 | Результаты сохраняются в машиночитаемом и человекочитаемом формате. |
| CMP-006 | Сравнение поддерживает несколько synthetic dataset одновременно. |
| CMP-007 | Метрики не должны обучаться и тестироваться на одинаковых сессиях. |

## Ошибки и восстановление

Несовместимая схема, отсутствующий обязательный отчёт валидации или пересечение обучающих и тестовых сессий останавливает сравнение до вычисления метрик. Ошибка отдельной реализации отмечается по контракту результата и не скрывает успешные результаты других метрик.

Повторный запуск использует неизменяемые пакеты, тот же профиль и явно сохранённое разбиение сессий. Частично записанные файлы не публикуются как готовый результат стадии.
