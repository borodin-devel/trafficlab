# Проверка синтетического трафика

## Назначение

Стадия проверяет последовательность packet events — абстрактных пакетных событий — и материализованный синтетический трафик. Она отделяет структурно и семантически допустимый результат генерации от кандидата, пригодного только для диагностики.

## Место в конвейере

Стадия выполняется после [генерации](../50-generate/README.md). Только успешный отчёт разрешает передать синтетический результат стадии сравнения.

## Входы

- `synthetic-manifest.json` по [контракту составного синтетического артефакта](../../contracts/packet-events.md#составной-синтетический-артефакт), через который доступны последовательность событий, материализованный Packet Capture Next Generation (PCAPNG) и, для active mode, захват со свидетельством изоляции;
- исходный запрос генерации;
- профиль проверки с физическими пределами, подмножеством канонического синтетического адресного пространства endpoint — конечных сетевых узлов — и настройками протоколов.

## Профили

Стадия поддерживает три явных профиля составной проверки:

| Профиль | Обязательные компоненты | Применимость |
| --- | --- | --- |
| `synthetic-events-v1` | `packet_events`, `dataset` | `rendered_pcapng` необязателен; правила байтов пакетов при отсутствии материализованного компонента получают `skip` с причиной `component_not_present`. |
| `synthetic-rendered-v1` | `packet_events`, `dataset`, `rendered_pcapng` | Правила байтов пакетов применимы к `rendered_pcapng`. |
| `synthetic-active-v1` | `packet_events`, `dataset`, `active_capture`, `isolation_evidence` | Правила байтов пакетов применимы к `active_capture`; правила сетевой границы применимы и не могут получить `skip`. |

Обязательность правила означает обязательность при применимом для него профиле и компоненте. Отсутствие компонента, обязательного для выбранного профиля, всегда даёт `fail`, а не `skip`. Каноническая матрица применимости находится в [правилах](rules.md).

## Выходы

Для каждого принятого входного offline-кандидата каталог `stages/60_validate_synthetic/` содержит `stage.json` и два отчёта по [контракту отчёта валидации](../../contracts/validation-report.md):

- `validation.json` использует выбранный составной профиль, а его `validated_artifact_id` равен `artifact_id` в `synthetic-manifest.json`;
- `dataset-validation.json` использует профиль `synthetic-dataset-v1`, а его `validated_artifact_id` равен `components.dataset.artifact_id`; отчёт проверяет пакет по единому [контракту набора данных](../../contracts/dataset.md) и согласованность его источника с компонентом `packet_events` того же манифеста.

Оба отчёта формируются независимо и сохраняются даже при `fail` одного из них. Неуспешный отчёт сохраняется для диагностики, но не разрешает сравнение. Проверяемые компоненты остаются неизменными.

## Основной процесс

Валидатор сначала проверяет манифест, требуемые выбранным профилем компоненты и их хеши. Затем он применяет каноническую [матрицу правил](rules.md) и параллельно проверяет набор данных для второго отчёта.

Несвязанные отдельные файлы не принимаются как эквивалент манифеста и его компонентов.

Профиль задаёт только параметры, явно разрешённые матрицей правил. Правило, неприменимое к выбранному транспортному протоколу или режиму генерации, получает `skip` с машиночитаемой причиной.

## Инварианты

- Каждое обязательное применимое правило представлено в отчёте как `pass` или `fail`.
- `skip` не скрывает отсутствующий обязательный вход.
- Проверка не исправляет события и не переписывает PCAPNG.
- Фактические числа потоков и сессий сверяются с тем же запросом, который породил результат.
- Повторная проверка одного артефакта одним профилем даёт тот же набор результатов.

## Ошибки и восстановление

Неподдерживаемая версия, невозможность прочитать обязательный вход или нарушение правила с фактической серьёзностью `error` дают статус `fail`. Предупреждения без ошибок дают `pass_with_warnings`. Внутренняя ошибка валидатора завершает стадию как `failed` и не подменяется отрицательным отчётом.
