# Эволюционный поиск

## Назначение

Стадия ищет конфигурации модели и генерации по нескольким целям качества. Она управляет популяциями кандидатов, повторно вызывает готовые стадии конвейера и сохраняет происхождение каждого результата.

## Место в конвейере

Стадия использует [обучение](../40-train/README.md), [генерацию](../50-generate/README.md), [проверку синтетического трафика](../60-validate-synthetic/README.md) и [сравнение](../70-compare/README.md) как повторно вызываемые операции. Это циклическая стадия поиска, а не обратное ребро в ориентированном ациклическом графе задач.

## Входы

- проверенный исходный [пакет набора данных](../../contracts/dataset.md);
- пространство поиска с типами моделей, диапазонами параметров и допустимыми генетическими операторами;
- ограничения допустимости кандидата и бюджет ресурсов;
- профиль целевых метрик по [контракту результата метрики](../../contracts/metric-result.md);
- условия завершения и фиксированные seed — исходные значения генераторов псевдослучайных чисел;
- совместимые версии контрактов модели, набора данных, проверки и метрик.

Генотип — каноническое описание наследуемых параметров кандидата. Он может включать тип и гиперпараметры модели, параметры признаков и генерации, протокольные ограничения, веса функции потерь, параметры агрегации метрик и политику seed. Необработанные веса нейронной сети в генотип не входят.

## Выходы

Стадия сохраняет состояние поиска, таблицу популяции и неизменяемый checkpoint — контрольную точку — каждого завершённого поколения в `stages/80_evolve/` по [контракту каталога запуска](../../contracts/run-directory.md). Контрольная точка содержит уникальные в пределах запуска и поколения `candidate_id`, генотипы, их хеши, оценки или причины штрафа, ранги, использованные seed, ссылки на артефакты вызванных стадий и полную genealogy — родословную кандидатов.

Победители каждого поколения публикуются в `champions/` как самостоятельные артефакты. Такой артефакт содержит генотип, происхождение, вектор качества и ссылки на неизменяемые модель, синтетический набор, отчёт проверки и результат сравнения. Он не зависит от изменяемого состояния текущей популяции.

## Основной процесс

Стадия создаёт или восстанавливает популяцию, вычисляет канонический genotype hash — хеш генотипа — и проверяет кэш. Для новых кандидатов она выполняет [цикл поиска](search-cycle.md), после чего передаёт допустимые векторы качества выбранному многокритериальному оптимизатору. Оптимизатор выбирает родителей и создаёт генотипы следующего поколения через mutation — мутацию — и crossover — скрещивание.

После завершения поколения стадия публикует его победителей и контрольную точку. [Правила оркестрации](../../common/orchestration.md) представляют поколения последовательностью сохранённых задач без циклических зависимостей между узлами.

## Инварианты

Cache — кэш — сопоставляет genotype hash с результатом только в том же контексте оценки: при одинаковых исходном наборе, разрешённой конфигурации, профилях, версиях реализаций и контрактов. Одинаковая каноническая сериализация генотипа даёт одинаковый хеш. Изменение любого поля генотипа даёт другой genotype hash; коллизия хеша считается ошибкой целостности. `candidate_id` идентифицирует экземпляр кандидата, а genotype hash служит идентичностью кэша и проверкой согласованности, поэтому разные кандидаты могут ссылаться на один уже рассчитанный генотип.

Genealogy хранит для кандидата `candidate_id`, поколение и checkpoint. Для каждого родителя она сохраняет `parent_candidate_id`, `parent_generation`, `parent_checkpoint` и `parent_genotype_hash` для проверки согласованности. Кроме того, запись содержит применённые операторы, их параметры и seed операции. Начальный кандидат отмечается как созданный без родителей. Genotype hash не используется как единственный идентификатор родителя; запись происхождения не переписывается после публикации.

В формулировках требований search space означает пространство поиска, а target metrics — целевые метрики. Порядок и имена компонентов `quality_vector` принадлежат только [контракту результата метрики](../../contracts/metric-result.md), а нормализация и отношение доминирования — [многокритериальной оценке стадии 70](../70-compare/evaluation.md). Стадия поиска передаёт этот вектор оптимизатору без переопределения его семантики.

| ID | Требование |
| --- | --- |
| EVO-001 | Оптимизатор получает search space, ограничения и целевые метрики. |
| EVO-002 | Каждый кандидат имеет уникальный `candidate_id` и канонический genotype hash. |
| EVO-003 | Уже рассчитанный кандидат повторно не запускается. |
| EVO-004 | Поддерживаются лимиты времени, поколений и целевого качества. |
| EVO-005 | Невалидные кандидаты получают штраф и не переходят дальше. |
| EVO-006 | Сохраняется полная genealogy — происхождение кандидатов. |
| EVO-007 | Поддерживается многокритериальная оптимизация. |
| EVO-008 | Победители каждого поколения сохраняются как самостоятельные артефакты. |

## Ошибки и восстановление

Кандидат с неуспешным обязательным отчётом проверки получает штраф допустимости; диагностика и это состояние сохраняются в checkpoint. Он не передаётся сравнению, ранжированию по Non-dominated Sorting Genetic Algorithm II (NSGA-II; по-русски — «многокритериальный генетический алгоритм с недоминируемой сортировкой»), отбору или размножению и не попадает в следующую популяцию. Штраф не записывается как вымышленный `quality_vector`. Если после исключения не осталось допустимых кандидатов, поколение завершается с ошибкой либо явно настроенная политика замещения создаёт новые кандидатские генотипы; недопустимые кандидаты никогда не становятся родителями. Ошибка инфраструктуры, отмена или исчерпание ресурсов не считается невалидностью генотипа и сохраняется отдельно для диагностики и политики повторов.

Поиск завершается при достижении `max_generations`, жёсткого дедлайна `max_wall_time`, заданных порогов качества, `no_improvement_generations`, ручной отмене или исчерпании ресурсов. Порог качества проверяет компоненты `quality_vector` по именам из его контракта либо `weighted_score`, если профиль задаёт его только как условие завершения; это не меняет отбор по Парето.

Возобновление начинается с последней полностью опубликованной контрольной точки. Кэш принимается только после проверки контекста, статуса и хешей артефактов. Незавершённое поколение восстанавливается из отдельных готовых задач либо вычисляется заново по правилам оркестрации.

## Предыдущая и следующая стадии

Стадия переиспользует стадии 40–70 и может начинаться после проверки исходного набора. Выбранный победитель является конечным результатом поиска либо входом отдельного запуска генерации. Следующей обязательной стадии нет.
