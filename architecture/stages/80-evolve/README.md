# Эволюционный поиск

## Назначение

Стадия ищет конфигурации модели и генерации по нескольким целям качества. Она управляет популяциями кандидатов, повторно вызывает готовые стадии конвейера и сохраняет происхождение каждого результата.

## Место в конвейере

Стадия использует [обучение](../40-train/README.md), [генерацию](../50-generate/README.md), [проверку синтетического трафика](../60-validate-synthetic/README.md) и [сравнение](../70-compare/README.md) как повторно вызываемые операции. Это циклическая стадия поиска, а не обратное ребро в ориентированном ациклическом графе задач.

## Входы

- проверенный исходный [пакет набора данных](../../contracts/dataset.md);
- пространство поиска с типами моделей, диапазонами параметров и допустимыми генетическими операторами;
- ограничения допустимости кандидата и бюджет ресурсов;
- профиль целевых метрик по [контракту результата метрики](../../contracts/metric-result.md);
- условия завершения и фиксированные seed — исходные значения генераторов псевдослучайных чисел;
- совместимые версии контрактов модели, набора данных, проверки и метрик.

Генотип — каноническое описание наследуемых параметров кандидата. Он может включать тип и гиперпараметры модели, параметры признаков и генерации, протокольные ограничения, веса функции потерь, параметры агрегации метрик и политику seed. Необработанные веса нейронной сети в генотип не входят.

## Выходы

Стадия сохраняет состояние поиска, таблицу популяции и неизменяемый checkpoint — контрольную точку — каждого завершённого поколения в `stages/80_evolve/` по [контракту каталога запуска](../../contracts/run-directory.md). Контрольная точка содержит генотипы, их хеши, оценки или причины штрафа, ранги, использованные seed, ссылки на артефакты вызванных стадий и полную genealogy — родословную кандидатов.

Победители каждого поколения публикуются в `champions/` как самостоятельные артефакты. Такой артефакт содержит генотип, происхождение, вектор качества и ссылки на неизменяемые модель, синтетический набор, отчёт проверки и результат сравнения. Он не зависит от изменяемого состояния текущей популяции.

## Основной процесс

Стадия создаёт или восстанавливает популяцию, вычисляет канонический genotype hash — хеш генотипа — и проверяет кэш. Для новых кандидатов она выполняет [цикл поиска](search-cycle.md), после чего передаёт допустимые векторы качества выбранному многокритериальному оптимизатору. Оптимизатор выбирает родителей и создаёт генотипы следующего поколения через mutation — мутацию — и crossover — скрещивание.

После завершения поколения стадия публикует его победителей и контрольную точку. [Правила оркестрации](../../common/orchestration.md) представляют поколения последовательностью сохранённых задач без циклических зависимостей между узлами.

## Инварианты

Cache — кэш — сопоставляет genotype hash с результатом только в том же контексте оценки: при одинаковых исходном наборе, разрешённой конфигурации, профилях, версиях реализаций и контрактов. Одинаковая каноническая сериализация генотипа даёт одинаковый хеш. Изменение любого поля генотипа даёт другой идентификатор кандидата; коллизия хеша считается ошибкой целостности.

Genealogy хранит идентификатор кандидата и поколения, genotype hash родителей, применённые операторы и их параметры, а также seed операции. Начальный кандидат отмечается как созданный без родителей. Запись происхождения не переписывается после публикации.

В формулировках требований search space означает пространство поиска, а target metrics — целевые метрики. Многокритериальная оптимизация использует `quality_vector` из результата сравнения. Вектор содержит оценки распределений, последовательностей, протоколов, потоков и сессий, а также классификационного двухвыборочного теста в порядке, заданном контрактом. `weighted_score` не заменяет этот вектор и не используется для доминирования по Парето.

| ID | Требование |
| --- | --- |
| EVO-001 | Оптимизатор получает search space, ограничения и целевые метрики. |
| EVO-002 | Каждый кандидат имеет уникальный genotype hash. |
| EVO-003 | Уже рассчитанный кандидат повторно не запускается. |
| EVO-004 | Поддерживаются лимиты времени, поколений и целевого качества. |
| EVO-005 | Невалидные кандидаты получают штраф и не переходят дальше. |
| EVO-006 | Сохраняется полная genealogy — происхождение кандидатов. |
| EVO-007 | Поддерживается многокритериальная оптимизация. |
| EVO-008 | Победители каждого поколения сохраняются как самостоятельные артефакты. |

## Ошибки и восстановление

Кандидат с неуспешным обязательным отчётом проверки получает штраф допустимости и не передаётся сравнению или размножению. Штраф не записывается как вымышленный `quality_vector`: оптимизатор ранжирует любой допустимый результат выше недопустимого. Ошибка инфраструктуры, отмена или исчерпание ресурсов не считается невалидностью генотипа и сохраняется отдельно для диагностики и политики повторов.

Поиск завершается при достижении `max_generations`, `max_wall_time`, заданных порогов качества, `no_improvement_generations`, ручной отмене или исчерпании ресурсов. Порог качества проверяет явно названные компоненты `quality_vector` либо `weighted_score`, если профиль задаёт его только как условие завершения; это не меняет отбор по Парето.

Возобновление начинается с последней полностью опубликованной контрольной точки. Кэш принимается только после проверки контекста, статуса и хешей артефактов. Незавершённое поколение восстанавливается из отдельных готовых задач либо вычисляется заново по правилам оркестрации.

## Предыдущая и следующая стадии

Стадия переиспользует стадии 40–70 и может начинаться после проверки исходного набора. Выбранный победитель является конечным результатом поиска либо входом отдельного запуска генерации. Следующей обязательной стадии нет.
