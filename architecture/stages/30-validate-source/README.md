# Проверка исходного набора

## Назначение

Стадия применяет правила профиля исходных данных к уже опубликованному пакету набора данных. Она отделяет пригодный вход обучения от повреждённого, несовместимого или допускающего утечку между частями набора.

## Место в конвейере

Стадия выполняется после [конвертации](../20-convert/README.md). Только успешный отчёт разрешает передать исходный набор стадии обучения.

## Входы

- неизменяемый пакет по [контракту набора данных](../../contracts/dataset.md);
- профиль проверки с порогами и уровнями серьёзности;
- ожидаемые хеши и сведения о происхождении результата конвертации.

## Выходы

Каталог `stages/30_validate_source/` содержит `stage.json` и отдельный `validation.json` уровня стадии в формате JavaScript Object Notation (JSON) по [контракту каталога запуска](../../contracts/run-directory.md). Файл `stages/30_validate_source/validation.json` использует [общий контракт валидации](../../contracts/validation-report.md), ссылается на проверенный идентификатор артефакта и содержит результат каждого правила профиля исходных данных.

Проверяемый пакет, включая созданный при публикации внутренний `dataset/validation.json` с базовой структурной проверкой, остаётся неизменным. Отчёт уровня стадии не заменяет и не перезаписывает этот файл. Неуспешный отчёт сохраняется как диагностический результат, но не разрешает использование набора следующей стадией.

## Основной процесс

Валидатор открывает манифест и схему, проверяет поддерживаемость версий, а затем последовательно читает таблицы Apache Parquet. Он собирает ограниченное состояние для проверки индексов, временных значений, ссылок и разделения сессий. После чтения валидатор сверяет агрегаты и хеши и формирует единый отчёт.

Канонический перечень и смысл проверок определены в [правилах исходного набора](rules.md). Профиль может менять порог и серьёзность только там, где это явно разрешено правилом; он не удаляет обязательные проверки.

Повторяемость результата задают [общие правила воспроизводимости](../../common/reproducibility.md).

## Инварианты

- Все обязательные правила представлены в отчёте как `pass`, `fail` или `skip`.
- Обязательное правило нельзя пропустить без машиночитаемой причины.
- Хеш проверяется для тех же неизменяемых файлов, которые прочитал валидатор.
- Ошибка уровня `error` запрещает передачу набора следующей стадии.

## Ошибки и восстановление

Неподдерживаемая версия, ошибка чтения либо нарушение правила с фактической серьёзностью `error` даёт статус `fail`; нарушение правила с серьёзностью `warning` при отсутствии ошибок даёт статус `pass_with_warnings`. Внутренняя ошибка валидатора завершает стадию как `failed` и не подменяется отрицательным результатом проверки. После исправления входа, профиля или реализации создаётся новое выполнение.
