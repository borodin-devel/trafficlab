# Платформа и ограничения

## Поддерживаемая среда

Первая версия запускает Linux-приложения внутри текущего дистрибутива подсистемы Windows для Linux версии 2 (Windows Subsystem for Linux version 2, WSL 2). Захват Windows-приложений и процессов другого WSL-дистрибутива не поддерживается.

Минимальная среда включает Python 3.12, `ip` из `iproute2`, `nft` из `nftables`, `dumpcap`, `capinfos`, `tshark`, `ethtool` и контрольные группы Linux версии 2 — control group version 2 (`cgroup v2`). Предварительная проверка (`preflight`) проверяет наличие конкретных команд и `/sys/fs/cgroup/cgroup.controllers`, а не названия пакетов дистрибутива.

## Сетевой уровень

Захват на виртуальном Ethernet-интерфейсе `veth` показывает Ethernet внутри Linux-среды WSL, а не физический Ethernet или Wi-Fi Windows. В этой системе канальный уровень (Layer 2, L2) означает длину кадра в точке захвата, тип протокола Ethernet, адреса виртуальной топологии, направление и временную последовательность.

Данные не описывают повторные передачи Wi-Fi, заголовки `radiotap`, физическую виртуальную локальную сеть (Virtual Local Area Network, VLAN), аппаратные временные метки, фактическое заполнение кадров Ethernet и поведение драйвера сетевого адаптера Windows.

Доменные сокеты Unix, каналы, именованные каналы «первым пришёл — первым вышел» (First In First Out, FIFO), разделяемая память и стандартные потоки не являются трафиком межсетевого протокола (Internet Protocol, IP) и не попадают в формат пакетного захвата нового поколения (Packet Capture Next Generation, PCAPNG). Интерфейс обратной петли (`loopback`) внутри целевого сетевого пространства имён (`namespace`) виден только при отдельном захвате этого интерфейса.

## Режим сети WSL

WSL может работать в режиме преобразования сетевых адресов (Network Address Translation, NAT) или в зеркальном сетевом режиме (`mirrored networking`). Режим определяется на предварительной проверке и сохраняется в метаданных запуска. Возможности межсетевого протокола версии 6 (Internet Protocol version 6, IPv6), виртуальной частной сети (Virtual Private Network, VPN), многоадресной рассылки (`multicast`) и доступа к Windows через локальный узел (`localhost`) зависят от режима; стадия не предполагает их наличие без проверки.

Туннелирование системы доменных имён (Domain Name System, DNS) в Windows 11 может обходить ожидаемую точку пакетного захвата. Для воспроизводимого захвата целевое сетевое пространство имён использует явный DNS-прокси-сервер (`proxy`) в основном сетевом пространстве имён WSL. Его адрес и фактическая конфигурация сохраняются в метаданных.

## Сетевая разгрузка и длина захвата

Перед захватом вспомогательная программа проверяет и по возможности отключает аппаратную разгрузку сетевой обработки (`offload`): сегментацию протокола управления передачей (Transmission Control Protocol Segmentation Offload, TSO), общую сегментацию (Generic Segmentation Offload, GSO), общее объединение принимаемых пакетов (Generic Receive Offload, GRO), объединение крупных принимаемых пакетов (Large Receive Offload, LRO), а также аппаратное вычисление контрольной суммы (`checksum offload`) при передаче (TX) и приёме (RX). Запрошенные и фактические значения сохраняются раздельно. Валидатор учитывает функции, которые отключить не удалось.

Рекомендуемая максимальная сохраняемая длина пакета (`snaplen`) — `256` байт. Она сохраняет заголовки, флаги и исходную длину, но сокращает полезную нагрузку (`payload`). Значение `snaplen = 0` разрешает полный пакет для специальных исследований.

## Ограничения масштаба

Первая версия не включает распределённое обучение, Kubernetes, Ray, Dask, Celery, Kafka и сервер MLflow. Фактический бюджет памяти определяется внутри WSL по [общей политике ресурсов](resources.md).

## Потребители

Ограничения учитывают предварительная проверка, захват, валидация, воспроизводимость, планировщик ресурсов и пользовательские отчёты.
