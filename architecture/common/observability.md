# Наблюдаемость

## Назначение

Наблюдаемость объединяет структурированные события, человекочитаемый журнал и измерения ресурсов. Она не является источником истины о готовности артефакта; готовность определяет [контракт стадии](../contracts/stage.md).

## Файлы

```text
logs/events.jsonl
logs/console.log
metrics/resource.parquet
```

JSON Lines (JSONL) хранит структурированные события. `console.log` предназначен для чтения человеком. `resource.parquet` хранит периодические измерения процессора, памяти и ввода-вывода.

Допустимые уровни:

```text
DEBUG
INFO
WARNING
ERROR
CRITICAL
```

## Архитектура журналирования

```text
worker process
    |
    | bounded queue
    v
logging listener process
    |
    +-> JSONL
    +-> console
```

Используется `QueueHandler`/`QueueListener` или эквивалент. Очередь ограничена. При переполнении первыми отбрасываются события `DEBUG`. События `ERROR` и `CRITICAL` имеют синхронный резервный путь записи.

## Правила событий и измерений

Пакеты не журналируются по одному. Компоненты записывают агрегаты через заданный интервал времени или число обработанных объектов. Процесс захвата сохраняет stderr в отдельный файл.

Каждое событие содержит временную метку Coordinated Universal Time (UTC), уровень, идентификаторы запуска и стадии, тип события и структурированные поля. Измерение ресурса содержит те же идентификаторы и монотонную временную привязку.

Секреты, payload и credentials из переменных окружения маскируются до помещения в очередь. Правила безопасности имеют приоритет над диагностической полнотой.

## Потребители

Данные создают все процессы. Их читают терминальный интерфейс, команды инспекции, диагностика сбоев, отчётность и анализ производительности.
