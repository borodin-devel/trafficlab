# Этапы разработки

Порядок снижает риск вертикально: сначала появляются устойчивые артефакты и проверки, затем захват и модели, после этого оркестрация, поиск и более дорогие режимы. Этап разработки не совпадает со стадией рабочего [конвейера](../system/pipeline.md).

## Этап 1. Контракты артефактов

Реализовать каталог запуска, манифест стадии, схему набора данных, отчёт проверки, хеширование и атомарную публикацию. Результат: система создаёт и проверяет пустые артефакты конвейера по [файловым контрактам](../contracts/).

## Этап 2. Преобразование и проверка

Поддержать внешний файл Packet Capture Next Generation (PCAPNG), декодирование через `tshark`, запись Apache Parquet, проверку через `capinfos` и экспорт JavaScript Object Notation Lines (JSONL) и Markdown. Результат: пользователь проверяет существующие дампы без собственного захвата.

## Этап 3. Случайная генерация и сравнение

Реализовать генератор случайных событий, построение пакетов через Scapy, проверку синтетического результата и базовые статистические метрики. Результат: часть конвейера сравнения тестируется до захвата и обучения моделей.

## Этап 4. Реализация захвата

Реализовать сетевое пространство имён, пару виртуальных Ethernet-интерфейсов, control group version 2 (cgroup v2), `dumpcap`, сквозную работу текстового пользовательского интерфейса — Text-based User Interface (TUI), очистку и Domain Name System (DNS). Результат: система получает реальный исходный набор данных по правилам [стадии захвата](../stages/10-capture/README.md).

## Этап 5. Базовые модели

Реализовать независимую эмпирическую, условную эмпирическую и полумарковскую модели. Результат: первый полный цикл захвата, обучения, генерации и сравнения. Общий интерфейс задают [стадия обучения](../stages/40-train/README.md) и [контракт модели](../contracts/model-artifact.md).

## Этап 6. Оркестратор

Реализовать ориентированный ациклический граф — Directed Acyclic Graph (DAG), частичные запуски, возобновление, кэш и бюджет ресурсов. Результат: стадии запускаются воспроизводимо и без ручной передачи путей. Политика принадлежит [оркестрации](../common/orchestration.md).

## Этап 7. Эволюционный поиск

Реализовать популяцию, отбор Парето, мутацию, контрольные точки и условия завершения. Результат: параметры моделей и генераторов оптимизируются по нескольким независимым целям согласно [циклу поиска](../stages/80-evolve/search-cycle.md).

## Этап 8. Активная эмуляция

Добавить активное сетевое взаимодействие только после стабилизации автономного синтеза. Результат: отдельный режим выполняется в изолированной среде и не меняет контракт автономного результата. Граница режимов описана в [документе генерации](../stages/50-generate/README.md).

## Этап 9. Нейронные модели

Сначала добавить авторегрессионную модель на основе Gated Recurrent Unit (GRU), затем ограниченную авторегрессионную модель Transformer. Результат: дорогие модели сравниваются с уже работающими базовыми решениями и используют тот же контракт артефакта.

Переход к следующему этапу требует работоспособного результата текущего этапа, его контрактных тестов и воспроизводимого минимального сценария. Этапы 8 и 9 не блокируют завершение первой практически полезной вертикали.
