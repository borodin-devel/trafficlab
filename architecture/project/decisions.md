# Архитектурные решения

Architecture Decision Record (ADR) — запись архитектурного решения. Все перечисленные решения приняты. Этот файл фиксирует выбор, основание и последствия; подробное устройство принадлежит связанному основному документу.

## ADR-001. Python 3.12

Выбор: единая поддерживаемая версия Python 3.12. Она соответствует целевой Ubuntu 24.04 и устраняет матрицу совместимости с Python 3.11. Версия и окружение фиксируются по правилам [зависимостей](dependencies.md).

## ADR-002. Модульный монолит

Выбор: один дистрибутив с явными внутренними модулями и несколькими консольными программами. Это упрощает локальную установку и повторное использование кода, а файловые контракты сохраняют границы процессов. Разбиение описано в [структуре репозитория](repository.md) и [модулях](../system/modules.md).

## ADR-003. Файловые контракты артефактов

Выбор: программы и стадии обмениваются версионируемыми файлами, а не общей памятью или неявными импортами. Это позволяет возобновлять частичные запуски, проверять происхождение и заменять реализацию стадии. Стабильные форматы находятся в [`contracts/`](../contracts/).

## ADR-004. Apache Parquet, JSON и TOML

Выбор: табличные данные хранятся в Apache Parquet, структурные манифесты — в JavaScript Object Notation (JSON), а пользовательские конфигурации — в Tom's Obvious Minimal Language (TOML). Форматы дают типизированные таблицы, читаемые метаданные и удобное редактирование конфигурации. Владелец формата набора — [контракт набора данных](../contracts/dataset.md).

## ADR-005. Захват в сетевом пространстве имён

Выбор: основная реализация использует Linux network namespace, пару виртуальных Ethernet-интерфейсов и `dumpcap`. Она даёт явную границу трафика приложения и наблюдаемую топологию. Ограничения и альтернативы описаны в [реализации захвата](../implementations/capture-backends/netns-dumpcap.md).

## ADR-006. Учёт дерева процессов через cgroup v2

Выбор: control group version 2 (cgroup v2) определяет всё дерево процессов запуска. Отслеживание одного process identifier (PID) недостаточно, потому что потомок может пережить родителя. Жизненным циклом владеет [управление процессами](../stages/10-capture/process-control.md).

## ADR-007. `dumpcap` для захвата

Выбор: `dumpcap` выполняет запись пакетов. Это сужает привилегированную поверхность и отделяет захват от декодирования. Готовность и остановка определены в [правилах записи](../stages/10-capture/packet-capture.md).

## ADR-008. `tshark` как первый декодер

Выбор: первая реализация декодирования использует `tshark`. Инструмент поддерживает нужные протоколы и структурированный вывод, оставаясь заменяемым адаптером. Детали находятся в [реализации декодера](../implementations/packet-decoders/tshark.md).

## ADR-009. Scapy как первый построитель пакетов

Выбор: первая реализация материализации событий использует Scapy. Она подходит для автономного синтеза и тестовых пакетов, но не становится частью контракта события. Детали находятся в [реализации построителя](../implementations/packet-renderers/scapy.md).

## ADR-010. Функциональное ядро и императивная оболочка

Выбор: чистые преобразования отделяются от файлов, процессов, сети и часов. Это повышает детерминизм и позволяет проверять основную логику без привилегированной среды. Правила кода находятся в [стиле программирования](code-style.md).

## ADR-011. Явный реестр реализаций

Выбор: модели, генераторы, метрики и другие заменяемые блоки регистрируются явно. Побочные эффекты импорта и singleton-реестры не используются. Семейства и их статусы перечислены в [каталоге реализаций](../implementations/README.md).

## ADR-012. Разделение автономного синтеза и активной эмуляции

Выбор: автономное построение артефакта и активное сетевое взаимодействие являются разными режимами с разными входами и мерами безопасности. Активный режим не является скрытым шагом автономного. Граница закреплена в [стадии генерации](../stages/50-generate/README.md).

## ADR-013. Многокритериальное сравнение

Выбор: система сохраняет вектор результатов метрик и отношение Парето, а не скрывает компромисс в одном числе. Скаляризация допустима только как явно версионируемая политика. Правила принадлежат [оценке сравнения](../stages/70-compare/evaluation.md).

## ADR-014. Локальный планировщик с учётом ресурсов

Выбор: локальный планировщик резервирует память, процессорные потоки и исключительные группы до запуска задачи. Это ограничивает конкуренцию тяжёлых задач на целевой машине. Бюджеты и инварианты определяет [управление ресурсами](../common/resources.md).

## ADR-015. Не сохранять полезную нагрузку по умолчанию

Выбор: набор данных и другие подготовленные для модели данные по умолчанию не сохраняют байты прикладной полезной нагрузки. Сырой PCAPNG, напротив, хранит пакет до настроенной длины `snaplen`, поэтому может содержать полезную нагрузку и защищается как чувствительный сырой артефакт. Политику защиты задаёт [безопасность](../common/security.md), а границу содержимого PCAPNG — параметры [захвата пакетов](../stages/10-capture/packet-capture.md#длина-захвата).

## ADR-016. Не использовать `pickle` для публичных артефактов

Выбор: Python `pickle` не является публичным или долговременным форматом. Он небезопасен для недоверенных данных и связывает артефакт с кодом Python. Модели публикуются через проверяемые форматы из [контракта модели](../contracts/model-artifact.md) и [сериализаторов](../implementations/artifact-serializers/json-parquet-npz.md).
