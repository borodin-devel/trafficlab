# Тестовая стратегия

Проверки разделены по границе отказа. Быстрые чистые тесты дают основную обратную связь, а внешние программы, привилегии и полный конвейер проверяются отдельными наборами.

## Модульные тесты

Модульные тесты покрывают чистые преобразования, проверку схем, хеши, ключи кэша, определение направления пакета, протокольное состояние, метрики и генетические операторы. Они не требуют сети, привилегий или внешних программ.

Для инвариантов преобразований применяется тестирование на основе свойств. Фиксированное исходное значение генератора должно воспроизводить результат.

## Контрактные тесты

Каждая подключаемая реализация проходит общий набор проверок:

```text
обучение создаёт корректный артефакт
генерация детерминирована при фиксированном исходном значении
некорректная конфигурация отклоняется
артефакт объявляет версию схемы
выход проходит базовый валидатор
метрика возвращает конечный результат
```

Набор выполняется для каждого зарегистрированного типа модели, генератора, метрики, валидатора и сериализатора. Контрактные тесты проверяют поведение интерфейса, а не внутреннее устройство реализации.

## Интеграционные тесты

Интеграционные тесты проверяют границы с внешними средствами и операционной системой:

- преобразование `tshark` в Apache Parquet;
- построение Packet Capture Next Generation (PCAPNG) из событий Scapy;
- первичную проверку через `capinfos`;
- создание сетевого пространства имён и control group version 2 (cgroup v2);
- время жизни дерева процессов и очистку по тайм-ауту;
- прокси Domain Name System (DNS);
- настройку аппаратных сетевых оптимизаций — offload.

Привилегированные проверки помечаются явно:

```python
@pytest.mark.privileged
@pytest.mark.wsl
```

Метка `wsl` означает Windows Subsystem for Linux (WSL). Обычный набор тестов не получает повышенные привилегии.

## Системные тесты

Минимальный набор целевых приложений включает клиентов Transmission Control Protocol (TCP), User Datagram Protocol (UDP), DNS и Hypertext Transfer Protocol (HTTP), сетевой клиент в дочернем процессе, демонизирующийся дочерний процесс, макет текстового пользовательского интерфейса — Text-based User Interface (TUI), и процесс, сопротивляющийся тайм-ауту.

Сценарии покрывают полный конвейер, частичные запуски, интерактивный захват, очистку по тайм-ауту и возобновление. Их ожидаемые артефакты определяют [контракты](../contracts/), а не копии структур в тестах.

## Фикстуры

Эталонные файлы должны быть маленькими и иметь понятное происхождение. Для Parquet сравниваются схема и семантическое содержимое, а не весь бинарный файл. Политика хранения задана в [документе ресурсов](assets.md).

Общая фикстура не должна скрывать существенную настройку сценария. Временный каталог создаётся на тест и удаляется даже после ошибки. Внешние программы проверяются на доступность до запуска теста; отсутствие опциональной программы даёт явный пропуск, а не ложный успех.

## Покрытие

Рекомендуемые пороги:

```text
contracts, runstore и validation: 90%
чистая логика моделей и метрик:   85%
orchestration:                    80%
```

Для привилегированного интеграционного кода покрытие не является единственным критерием. Обязательны проверки успешного пути, частично созданных ресурсов, повторной очистки, сигналов и тайм-аутов.
