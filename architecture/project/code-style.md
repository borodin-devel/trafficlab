# Стиль программирования

Код следует принципу «функциональное ядро, императивная оболочка» — functional core, imperative shell. Решения и преобразования остаются чистыми, а внешние эффекты сосредоточены в тонких адаптерах.

## Функциональное ядро

Типичные чистые преобразования:

```text
поля пакета -> PacketEvent
пакет PacketEvent -> FlowTable
FlowTable -> FeatureTable
признаки исходного и синтетического наборов -> MetricResult
генотип + исходное значение генератора -> конфигурация кандидата
```

У функции явные входы и выходы. Она не читает глобальную конфигурацию, не выполняет скрытый ввод-вывод и при фиксированном исходном значении генератора даёт детерминированный результат. Такое ядро проверяется обычными и основанными на свойствах тестами.

## Императивная оболочка

Оболочка владеет файловой системой, дочерними процессами, сигналами, сетевыми пространствами имён, control group, журналированием, часами, созданием исходных значений генераторов и планированием процессов. Она переводит внешние ошибки в типизированные ошибки контракта и передаёт ядру явные значения.

Ресурс открывается и закрывается в одной видимой области. Изменяющая систему операция должна иметь идемпотентную очистку. Правила оркестрации и привилегий определены в [общих документах](../common/orchestration.md).

## Допустимые классы

Класс используется, когда он представляет одно из следующего:

- неизменяемую `dataclass`-структуру;
- Pydantic-модель входной границы;
- структурный интерфейс `Protocol`;
- обёртку модели, если внешняя библиотека требует объектный прикладной программный интерфейс — Application Programming Interface (API);
- контекстный менеджер ресурса;
- конечный автомат с явным набором состояний.

## Когда класс не нужен

Не создаются пространства имён из статических методов, классы `Manager`, `Factory` или `Service` без состояния, иерархии валидаторов, метрик или типов моделей и singleton-реестры. По умолчанию используются композиция, функции и `Protocol`, а не наследование.

Явный реестр реализаций хранит имя, фабрику и совместимость. Импорт модуля не должен неявно менять глобальный реестр.

## Границы и размер изменений

- Публичная межпрограммная граница остаётся файловой и следует [контрактам](../contracts/).
- Библиотечный модуль не печатает пользовательский текст и не завершает процесс.
- Адаптер интерфейса командной строки разбирает аргументы, вызывает библиотечную функцию и отображает результат.
- Небольшие модули и зеркальные тесты предпочтительны: это упрощает проверку человеком и небольшой языковой моделью.
- Новый абстрактный слой вводится только для уже существующей заменяемости, а не для предполагаемого будущего варианта.
